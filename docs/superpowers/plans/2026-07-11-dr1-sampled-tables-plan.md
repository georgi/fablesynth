# DR-1 Sample-Derived Drum Tables — Implementation Plan

## Goal

Ship 5 new factory wavetables for DR-1 derived from real CC0 TR-808 recordings
(snare, clap, closed hat, open hat, cymbal), so pads can select punchy
sample-flavored timbres via `oscA.table` / `oscB.table` in **both** the web
engine and the JUCE plugin, with byte-identical table data across the two.

## Source material (CC0-1.0, public domain)

From `tidalcycles/sounds-tr808-fischer` (real Roland TR-808 SN 103852, CC0-1.0).
Download via raw.githubusercontent.com `master` branch. Chosen representative
mid-position samples:

| Table name | Repo path            | Sound              |
|------------|----------------------|--------------------|
| `808SD`    | `sd8/SD5050.WAV`     | Snare (mid/mid)    |
| `808CP`    | `cp8/CP.WAV`         | Hand clap          |
| `808CH`    | `ch8/CH.WAV`         | Closed hat         |
| `808OH`    | `oh8/OH50.WAV`       | Open hat (mid dec) |
| `808CY`    | `cy8/CY5050.WAV`     | Cymbal (mid/mid)   |

Commit the 5 WAVs under `assets/drum-samples/808/` plus a `SOURCES.md` recording
provenance + CC0 license text/link. ~5 small 16-bit/44.1k mono files, well
under 1 MB total.

## Architecture / key facts (verified in code)

- Table format: a table = `FRAMES` (=16) time-domain single cycles of
  `SIZE` (=2048) samples, fed through `buildUserTable(name, frames)`
  (`src/engine/wavetables.ts:246`) which does the band-limit + 11-level mip
  build. **Both engines call the identical `buildUserTable`** — TS and a 1:1
  C++ port — so if both are handed the same 16×2048 frame floats, the
  resulting playable tables are identical by construction. `buildUserTable`
  accepts any frame count (`nf = frames.length`).
- Table array assembly, web: `drum-synth.ts:91`
  `builtInTables = [...generateDrumTables(), ...generateTables()]`
  → 4 procedural drum tables (THUD CRACK TINE GRIT) then 6 WT-1 procedurals
  (PRIME BLOOM PULSE VOX CHIME GLITCH) = 10, matching
  `DRUM_TABLE_NAMES` (`src/drum/params.ts:14`).
- Table array assembly, JUCE: `DrumProcessor.cpp:26-29` (same two loops);
  names in `DrumParams.cpp:9` `DRUM_TABLE_NAMES`.
- **Append new tables at the END** (index 10..14) so existing kit/preset
  table indices (0..9) never shift. Append the 5 names after `GLITCH` in
  both name lists.
- `pos` is a live mod destination in the drum worklet
  (`worklet-drum.js:157`), so a pos envelope sweeps frames 0→15 = the
  sample's spectral evolution over time.

## Converter design (offline Node script, run manually; output committed)

`scripts/build-drum-tables.mjs`:
1. Decode each WAV → mono Float32, normalize peak to 1.0. (16-bit PCM,
   44.1k, RIFF; write a tiny WAV parser — no deps.)
2. Pick 16 analysis centers across the sound's active region, spaced denser
   near the attack (exponential spacing over the length above a silence
   threshold).
3. For each center: Hann-window a 2048-sample chunk, real FFT, take
   per-harmonic **magnitude** (bins 1..1024). Build one 2048-sample cycle by
   assigning harmonic k that magnitude with a **fixed deterministic phase**
   (seeded PRNG, same phase for harmonic k across ALL frames and all tables →
   coherent inter-frame interpolation, no comb-filtering on pos sweep).
   IFFT → time frame. Normalize the frame.
4. Emit ONE source of truth per table (16×2048 Float32), then serialize to:
   - `src/drum/engine/sampledtables.gen.ts` — base64 Float32 frames (same
     encoding `usertables.ts` already uses) + a `generateSampledDrumTables()`
     that decodes → `buildUserTable`.
   - `juce/source/drum/dsp/SampledTables.gen.cpp` + `.h` — the same floats as
     a C array + `generateSampledDrumTables()` that wraps `buildUserTable`.
   Both files carry a header comment: generated by this script, do not edit.

Known limitation (accepted): magnitude-only resynthesis renders noisy sounds
(clap wash, hat sizzle) as spectrally-correct but tonal — the per-pad noise
layer supplies the stochastic component. 808 clap/hats are fairly tonal to
begin with, so this holds up. Min-phase reconstruction is a possible future
improvement, out of scope here.

## Global Constraints

- Do NOT change the table format, `buildUserTable`, or existing table
  indices 0..9. New tables append at index 10..14 in both engines.
- The 5 new names, in this exact order, appended after `GLITCH`:
  `808SD`, `808CP`, `808CH`, `808OH`, `808CY`.
- Frame data in the TS `.gen` and the C++ `.gen` MUST come from the same
  generator run (byte-identical floats). Never hand-edit generated files.
- All samples strictly CC0; provenance recorded in
  `assets/drum-samples/808/SOURCES.md`.
- No UI changes; tables appear automatically in the existing table selector.
- No new factory kit (out of scope).
